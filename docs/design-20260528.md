## Problem Statement

用户打算做多个小工具（知识采集、面试复盘、AI 总结、数据处理等），但不想每个工具单独起一个项目。需要一个"个人后台容器"，工具可以像插件一样接入，统一管理。第一个落地工具是知识采集——从中文付费内容网页（如知识星球、小报童、语雀等）保存内容到个人知识库。

## What Makes This Cool

不是"又一个后台管理系统"，而是一个**可扩展的个人工具平台**。每加一个新工具，只需要写一个 NestJS Module + 对应的 Vue 页面，然后注册进系统。菜单自动出现、任务自动可追踪、日志统一可见。第一版用知识采集工具验证插件架构，之后加工具就像搭积木。

简历上的表述应该是："设计并实现了一个插件化个人后台容器系统，支持工具模块动态接入、统一权限与菜单管理、统一任务调度与日志追踪"——而不是"做了一个网页采集工具"。

## Constraints

- MVP 阶段单用户，暂不做多用户/登录
- 只采集用户自己有权限访问的内容，不做绕过付费墙
- 个人学习与归档用途，不做公开分享
- 必须部署上线（公网可访问），作为简历亮点
- 技术栈：Vue 3 生态 + NestJS
- 工具间互调和流程串联是 v2 需求，MVP 中每个工具独立运行

## Premises

1. **平台优先于工具：** 先搭容器框架（路由、菜单、工具注册机制），再做第一个工具 —— 同意
2. **服务端采集优于浏览器扩展：** 用 Playwright 服务端渲染页面后提取内容，绕过客户端复制保护 —— 同意
3. **单用户先行：** MVP 不做登录/多用户，以后有需要再加 —— 同意
4. **部署上线是 MVP 的一部分：** 项目必须部署到公网可访问的 URL —— 同意

## Cross-Model Perspective

以下来自 ChatGPT（独立第二意见）：

**最酷版本：** 不是"个人后台+爬虫"，而是一个"个人 AI 工具操作系统"。工具之间可以互相调用——知识采集 → AI 总结 → 推送到面试知识库，从后台管理系统升级为"个人生产力中台"。

**核心洞察：** "用户想做的是一个工具平台，不是单一工具"——他真正想要的是一个可以持续扩展的个人技术资产。简历价值在于"设计了一个能不断长出工具的平台"。

**可借力的开源项目：**
- 后台容器参考：Appsmith、ToolJet（偷思路，不直接用）
- 工作流参考：n8n、Activepieces（了解工具串联模式）
- 采集工具：Playwright + Mozilla Readability + Crawlee

**周末 MVP 建议：** NestJS + Next.js（原文建议，但我们选 Vue）分层架构，做 4 个模块：容器壳子、工具注册协议、知识采集工具、任务中心。明确跳过：浏览器扩展、多用户权限、可视化工作流、AI 问答。

## Approaches Considered

### Approach A: 极速 MVP（Nuxt 3 全栈单体）
- 技术栈：Nuxt 3 + Prisma + SQLite + Element Plus
- 工时：S | 风险：低
- 优点：最快上线，部署简单
- 缺点：前后端耦合，后期扩展累
- 结论：放弃——架构太弱，简历价值不够

### Approach B: 理想架构（NestJS + Vue 3 分层）✅ 选中
- 技术栈：NestJS + Vue 3 + Vite + Pinia + Element Plus + PostgreSQL + Prisma + Redis + BullMQ + Playwright + Docker Compose
- 工时：M | 风险：中
- 优点：简历含金量最高，架构清晰，工具注册协议天然映射到 NestJS Module，BullMQ 做任务队列专业
- 缺点：环境搭建比方案 C 多 2-3 小时，Windows 上 Redis 需要 Docker/WSL
- 结论：选中——匹配"简历亮点+练习 SDD"的目标

### Approach C: 中间路线（Nuxt 3 + 轻量分层）
- 技术栈：Nuxt 3 + Prisma + Element Plus + Pinia + Playwright
- 工时：S-M | 风险：低-中
- 优点：部署简单，一个容器搞定
- 缺点：不如方案 B 正式，面试时分量轻
- 结论：放弃——用户偏好更"正式"的架构

**工具注册协议（第一版静态注册）：**

每个工具 = 后端一个 NestJS Module + 前端一个目录 + manifest 声明：

```ts
// manifest.ts
export const knowledgeCaptureManifest = {
  key: 'knowledge-capture',
  name: '知识采集',
  icon: 'DocumentCopy',
  route: '/tools/knowledge-capture',
}
```

**后端工具注册机制：** 

后端 manifest 结构与前端一致但额外包含 NestJS Module 引用和 BullMQ 处理器：

```ts
// server/src/tools/knowledge-capture/manifest.ts
export const manifest: ToolManifest = {
  key: 'knowledge-capture',
  name: '知识采集',
  icon: 'DocumentCopy',
  route: '/tools/knowledge-capture',
  module: KnowledgeCaptureModule,   // NestJS Module 类引用
  processors: [captureProcessor],   // BullMQ 任务处理器
}
```

每个工具 Module 在 `onModuleInit` 中调用 `ToolRegistry.register(manifest)`。ToolRegistry 将工具信息持久化到 `Tool` 表（首次注册时 insert，已存在则 skip），并注册 BullMQ 处理器。`GET /api/tools` 从 `Tool` 表读取当前启用的工具列表，而非内存——这样前端菜单始终与数据库状态一致。

**前端菜单生成机制：** 使用 Vite 的 `import.meta.glob` 在构建时扫描 `/tools/*/manifest.ts`：

```ts
// router/index.ts
const toolModules = import.meta.glob('../tools/*/manifest.ts', { eager: true })

const toolRoutes = Object.values(toolModules).map((mod: any) => ({
  path: mod.manifest.route,
  name: mod.manifest.key,
  component: () => import(`../tools/${mod.manifest.key}/index.vue`), // 懒加载
  meta: { title: mod.manifest.name, icon: mod.manifest.icon }
}))

// 合并进 router
toolRoutes.forEach(route => router.addRoute('layout', route))
```

侧边栏菜单从 `router.getRoutes()` 中读取 `meta` 属性动态渲染，不硬编码任何工具条目。

## Data Model

```prisma
// 工具注册表（平台级）
model Tool {
  id        Int      @id @default(autoincrement())
  key       String   @unique    // e.g. "knowledge-capture"
  name      String              // e.g. "知识采集"
  icon      String              // Element Plus icon name
  route     String              // e.g. "/tools/knowledge-capture"
  enabled   Boolean  @default(true)
  createdAt DateTime @default(now())
}

// 任务记录（平台级，所有工具共用）
model Job {
  id        Int      @id @default(autoincrement())
  toolKey   String              // 关联到 Tool.key
  status    String   @default("pending")  // pending | running | success | failed
  input     Json?               // 工具自定义入参（e.g. { url: "..." }）
  output    Json?               // 工具自定义结果（e.g. { itemId: 1 }）
  error     String?             // 失败时的错误信息
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

// 知识条目（知识采集工具专用，每个工具可定义自己的数据模型）
model KnowledgeItem {
  id              Int      @id @default(autoincrement())
  title           String
  url             String
  source          String?           // 来源平台名称，e.g. "知识星球"
  contentHtml     String?           // Readability 提取的原始 HTML
  contentMarkdown String?           // 转换后的 Markdown
  status          String   @default("draft")  // draft | published | archived
  capturedAt      DateTime @default(now())
  jobId           Int?              // 关联到采集任务
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt
}
```

## API Contract

### 平台级端点

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/tools` | 返回已注册且 enabled=true 的工具列表 |
| GET | `/api/jobs?toolKey=&status=&page=1&pageSize=20` | 分页查询任务列表，支持按工具和状态过滤 |
| POST | `/api/jobs/:id/retry` | 将失败任务重新入队 |

### 知识采集工具端点

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/tools/knowledge-capture/capture` | 创建采集任务：`{ url: string }` → 返回 `{ jobId: number }` |
| GET | `/api/tools/knowledge-capture/items?page=1&pageSize=20` | 分页查询已采集的知识条目列表 |
| GET | `/api/tools/knowledge-capture/items/:id` | 查看单条知识详情（含 Markdown 正文） |
| DELETE | `/api/tools/knowledge-capture/items/:id` | 删除一条知识 |

## 任务系统（BullMQ）

### 架构

```
Tool Module → POST /api/tools/xxx → Controller
  → 1. 写入 Job 表 (status=pending)
  → 2. 提交 BullMQ（携带 jobRecordId）
  → 3. Processor 执行，完成后通过全局事件监听器更新 Job 表状态
```

### 设计决策：DB 优先，BullMQ 为执行层

任务的状态始终以 `Job` 表为准。BullMQ 只负责异步执行——不承担状态持久化职责。

- **创建任务：** Controller 先在 `Job` 表中插入一条记录（`status=pending`），获得 `jobId`，然后将 `{ jobRecordId }` 作为 BullMQ job data 入队
- **状态同步：** 全局 BullMQ 事件监听器（`BullModule.registerQueue` 的 `@OnQueueEvent`）监听 `completed`/`failed`/`active` 事件，根据 `jobRecordId` 更新对应 `Job` 记录的状态
- **双写失败处理：** 如果 BullMQ enqueue 失败（Redis 不可用），DB 中的 Job 记录保持 `status=pending`，由定时任务（每 5 分钟）扫描超过 10 分钟仍未执行的 pending 记录，标记为 `failed` 并记录错误原因
- **失败重试：** BullMQ 内置指数退避（最多 3 次），3 次后标记 `failed`。前端任务中心提供手动重试按钮（`POST /api/jobs/:id/retry`）

### 错误分类

采集任务可能的失败类型及处理：

| 错误类型 | 原因 | Job.status | 重试？ |
|---------|------|-----------|--------|
| NETWORK_ERROR | URL 不可达、DNS 解析失败 | failed | 自动重试 1 次 |
| TIMEOUT | 页面加载超过 30s | failed | 自动重试 1 次 |
| EXTRACTION_FAILED | Readability 未能提取正文 | failed | 不重试（需要人工判断） |
| BLOCKED | 页面检测到机器人、返回验证码 | failed | 不重试 |
| BROWSER_CRASH | Chromium 进程崩溃 | failed | 自动重试 2 次 |
| EMPTY_CONTENT | 提取到的正文为空或仅含导航文本 | failed | 不重试 |

## Docker & 部署说明

### Docker Compose 服务（4 个容器）

```

redis     — Redis 7，BullMQ 依赖
server    — NestJS（port 3000），含 Playwright 浏览器
client    — Vue 3 静态文件，Nginx 托管（port 80）
```

### Playwright in Docker

NestJS 的 Dockerfile 需要安装 Chromium 系统依赖：

```dockerfile
FROM node:22-slim
RUN apt-get update && apt-get install -y \
  libnss3 libnspr4 libatk1.0-0 libatk-bridge2.0-0 \
  libcups2 libdrm2 libdbus-1-3 libxkbcommon0 \
  libxcomposite1 libxdamage1 libxfixes3 libxrandr2 \
  libgbm1 libpango-1.0-0 libcairo2 libasound2t64 \
  && rm -rf /var/lib/apt/lists/*
# ... rest of build
RUN npx playwright install chromium --with-deps
```

### 部署资源建议

Chromium 实例在采集任务运行时会占用 500MB-1.5GB 内存。建议 VPS 最低配置 2GB RAM + 2GB swap。Playwright 启动时使用 `--disable-dev-shm-usage` 标志避免 `/dev/shm` 不足导致崩溃。如果内存紧张，可以限制并发采集任务数为 1。

Windows 本地开发时，Redis 和 PostgreSQL 通过 Docker Desktop 运行，NestJS 和 Vue 前端在宿主机直接跑（不需要进容器）。也可以用 WSL2 替代 Docker Desktop。
