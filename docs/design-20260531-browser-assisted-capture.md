## Problem Statement

当前知识采集需要用户手动分析目标网站的登录态——Cookie 在哪个 key、localStorage 里存了什么 token——然后 F12 复制粘贴到采集页面。流程繁琐、容易出错、且对非技术用户不友好。

目标：在用户已合法登录并能访问目标页面的前提下，消除手动复制 Cookie/Token 的步骤。不绕过登录和付费墙控制，不做公开分发，仅用于个人归档。

## What Makes This Cool

不是"又一个爬虫工具"。用户浏览网页时看到一篇想保存的文章，点一下浏览器扩展按钮就完成了——不需要离开页面、不需要 DevTools、不需要理解什么是 Cookie vs localStorage。扩展自动提取当前页面的完整登录上下文，发给 super-admin 后端，后端用 Playwright 以相同身份渲染页面、提取正文、转 Markdown、入库。用户只需要浏览和点击。

这个架构的独特之处：**浏览器扩展只做一件事——桥接登录态**。不做内容提取（留给服务端 Playwright），不做页面解析（留给 Readability）。扩展约 100 行代码（service-worker.js ~45 行 + content-script.js ~20 行 + popup.html ~50 行）。简洁，安全，可审计。

## Constraints

- 合法登录前提：用户必须已在浏览器中登录目标网站
- 不绕过付费墙：只采集用户有权限访问的内容
- 个人归档用途：不做公开分享
- MVP 只做 Chrome（Manifest V3），Firefox/Safari 后续
- 扩展通过开发者模式加载，不发布 Chrome Web Store（MVP 阶段）
- 架构设计支持未来分发给其他用户
- 保持现有后端采集管线不变（Playwright → Readability → Turndown → DB）

## Premises

1. **浏览器扩展是"免手动复制"的最直接方案**——Playwright persistentContext 连接本地 Chrome profile 是备选，但用户体验不如扩展 —— 同意
2. **服务端 Playwright 渲染仍然需要保留**——浏览器扩展只负责获取登录态，不做内容提取 —— 同意
3. **第一版只做 Chrome 扩展**——个人使用场景在 Windows 桌面端，Chrome 覆盖面足够 —— 同意
4. **不改变现有后端采集管线**——前端采集方式的变化不影响 Playwright → Readability → Turndown → DB 流程 —— 同意
5. **扩展设计上支持未来分发**——架构不硬编码单人假设，但 MVP 通过开发者模式加载 —— 同意

## Approaches Considered

### Approach A: 浏览器扩展 + 服务端 Playwright ✅ 选中

Summary: Chrome 扩展自动读取当前页面的 Cookie 和 localStorage，通过 HTTP 请求发送给 super-admin 后端。后端以相同登录态启动 Playwright 渲染页面、提取正文、转 Markdown 入库。扩展保持极简——只做"获取登录态 + 发送 URL"这一件事。

Effort: M（扩展开发 + 后端小适配）
Risk: 低——核心采集管线不变，扩展功能边界清晰

Pros:

- 用户体验最简：浏览到目标页 → 点扩展按钮 → 完成，无需离开页面
- 架构清晰：扩展 = 轻量登录态桥接，后端 = 重渲染引擎，各司其职
- 扩展代码极简（~100 行），安全可审计
- 后端采集管线（Playwright → Readability → Turndown → DB）全部复用，仅需新增 API Token 机制
- 天然支持未来分发——标准 Chrome 扩展格式

Cons:

- 需要开发 Chrome 扩展（新代码库：MV3 + service worker）
- 不能采集"当前没在看的页面"——需要先导航到目标页
- 本地开发需要同时跑 docker compose + 扩展 + 前端

### Approach B: 本地 Playwright persistentContext

Summary: 服务端 Playwright 复用本地 Chrome 用户数据目录，自动继承浏览器登录态。

Effort: S（后端小改）
Risk: 中——Chrome profile 锁定、跨平台路径差异、headless 兼容性

Pros:

- 改动极小：只改 capture.processor.ts 的 browser 启动方式
- 不需要浏览器扩展，不需要新代码库

Cons:

- 操作步骤不减：仍需动命令行/关浏览器来避免 profile 锁定
- 跨机器不可移植，服务器部署后无法使用
- 本质上是把"复制粘贴"换成"配置 Chrome 路径"——用户体验改善有限

### Approach C: bookmarklet 渐进改进

Summary: 提供浏览器书签小工具一键复制登录态到剪贴板，减少 F12 步骤。

Effort: XS（前端小改 + bookmarklet）
Risk: 极低——零架构变化

Pros:

- 最快落地，跨浏览器通用

Cons:

- 仍然需要手动操作（只是从 F12 → 点书签 → 粘贴）
- 不从根本上解决问题
- 不解决"未来分发给其他用户"的场景

## Recommended Approach

**方案 A：浏览器扩展 + 服务端 Playwright 混合架构**

### 通信协议

**Request:**

```
POST /api/tools/knowledge-capture/capture
Content-Type: application/json
Authorization: Bearer <api-token>

{
  "url": "https://xiaobot.net/post/xxx",
  "cookies": "[{\"name\":\"session\",\"value\":\"abc123\",\"domain\":\".xiaobot.net\"}]",
  "localStorage": "{\"vuex\":\"{...}\"}"
}
```

注意：`cookies` 和 `localStorage` 字段保持 JSON 字符串格式，与现有的 CapturePage.vue 发送格式完全一致。扩展 service-worker 在 POST 前做 `JSON.stringify()`。后端 Controller 无需任何改动。

**Response (201 Created):**

```json
{ "jobId": 42 }
```

**Response (400 Bad Request):**

```json
{ "statusCode": 400, "message": "Invalid URL format" }
```

**Response (401 Unauthorized):**

```json
{ "statusCode": 401, "message": "Invalid API token" }
```

**Response (500 Internal Server Error):**

```json
{ "statusCode": 500, "message": "Capture failed: NETWORK_ERROR" }
```

### 鉴权模型

MVP 采用**共享密钥模式**——不是用户鉴权，是服务鉴权。Token 是一个数据库记录，用于证明"调用方持有合法采集凭证"。

- Token 首次启动时自动生成，存入 `ApiToken` 表（SHA-256 哈希），控制台打印原始值
- 后续启动不重新生成
- Web 设置页每次"授权此浏览器"时：生成新 raw token → SHA-256 存 DB（覆盖旧值）→ 返回原始值 → 前端外发给扩展
- 旧 token 立即失效（MVP 只有一个扩展，可接受）
- 数据库重置后需重新授权
- **FUTURE: 多用户支持时，ApiToken 表添加 `userId` 字段关联 User 表；setup-key 改为列表管理（生成/吊销/命名）**

### Chrome 扩展结构

```
extension/
├── manifest.json          # MV3, permissions: activeTab, storage, cookies, scripting
├── popup.html             # 内联 CSS + HTML：采集按钮 / 未配置引导
├── service-worker.js      # 核心调度：提取 cookies → 请求 localStorage → POST 后端
├── content-script.js      # 页面注入：序列化当前页面的 localStorage
└── icons/
    └── browser-plugin-icon.svg  # 复用 client/public/browser-plugin-icon.svg
```

关键简化：

- **砍掉 popup.js**——popup.html 内联 `<script>` 处理 UI 交互（代码量极少）
- **砍掉 sessionStorage 提取**——MVP 只提取 localStorage + Cookie。绝大多数网站 auth token 不放在 sessionStorage
- **砍掉 popup 设置面板**——配置由 Web 前端通过 `chrome.runtime.sendMessage` 直接推送
- **砍掉扩展内采集结果轮询/通知**——popup 提交后显示"查看任务"链接指向 JobCenter 页面

各模块职责：

- **popup.html** — 极简 UI。未配置时显示"前往 super-admin 设置页授权"按钮；已配置时显示采集按钮 + 当前 URL + loading/成功/错误状态 + "查看任务"链接。5 秒后自动关闭。
- **service-worker.js** — 核心调度。监听两类消息：① popup 的 collect 请求 → `chrome.cookies.getAll({url})` → `chrome.tabs.sendMessage` 请求 localStorage → 汇总序列化 → `fetch(POST /capture)` → 返回结果给 popup。② Web 前端的配置推送（`chrome.runtime.onMessageExternal`）→ 写入 `chrome.storage.local`。
- **content-script.js** — 仅做一件事：序列化当前页面的 localStorage。通过 `chrome.scripting.executeScript` 按需注入（`runAt: 'document_idle'`），收到消息后返回存储快照。

  **localStorage 序列化陷阱：** `JSON.stringify(localStorage)` 在 Chrome 中直接返回 `"{}"` ——因为 `localStorage` 是 `Storage` 接口对象，不是普通 JS 对象。正确的提取方式：

  ```js
  function serializeStorage(storage) {
    const data = {};
    for (let i = 0; i < storage.length; i++) {
      const key = storage.key(i);
      data[key] = storage.getItem(key);
    }
    return data;
  }
  const lsData = serializeStorage(localStorage);
  ```

### Web 前端 → 扩展 配置推送流

Web 前端 `/settings` 页面通过 `chrome.runtime.sendMessage` 直接推送 API Token 到扩展，**零复制粘贴**：

```
super-admin /settings 页面
  │
  │ 用户点击"授权此浏览器"
  │
  │──GET /api/auth/token──→ backend（生成 raw token + SHA-256 入库）
  │←── { token: "..." } ──
  │
  │──chrome.runtime.sendMessage(extId, { action: 'setConfig', token, backendUrl })──→ service-worker.js
  │                                                                                    │
  │                                                                                    │──chrome.storage.local.set({...})
  │                                                                                    │
  │←── { success: true } ──────────────────────────────────────────────────────────────
  │
  页面显示：✅ 扩展已授权
```

前提条件：

- 扩展 ID 固定（通过 `manifest.json` 的 `"key"` 字段——从 PEM 私钥推算）
- `manifest.json` 的 `externally_connectable.matches` 包含 Web 前端 origin
- Web 前端 `.env` 配置 `VITE_EXTENSION_ID=<固定ID>`

### 关键设计决策

1. **权限模型：`activeTab` + `cookies`**（Step 0 Spike 验证）
   - 在写任何代码之前先做验证：创建最小 MV3 扩展，仅声明 `activeTab` + `cookies` 权限，在目标网站点击 popup 调用 `chrome.cookies.getAll({url: tab.url})`
   - 若 `activeTab` 不足，回退为声明 `<all_urls>` 主机权限
   - 参见 Next Steps #0

2. **localStorage 的 origin 边界**
   - `content-script.js` 注入当前页面后，`localStorage` API 读取的是当前页面 origin 的数据
   - 若 auth token 存储在子域（如 `accounts.xiaobot.net` 的 localStorage），扩展提取不到
   - 影响评估：绝大多数网站的 auth token 在 Cookie 中（可跨子域），少数 SPA（如 xiaobot.net）把 token 放在 localStorage 中但属于同一 origin。此限制在现有已知目标网站中不影响使用

3. **扩展配置——Web 推送而非手动输入**
   - API Token 不存储在浏览器页面的 localStorage 中，也不存储在 `.env` 文件中共享给前端
   - 流程：Web 前端调 `GET /api/auth/token` 获取 → 通过 `chrome.runtime.sendMessage` 推送给扩展 → 扩展存入 `chrome.storage.local`
   - 用户全程只需点击一个按钮。无需输入框，无需复制粘贴
   - 后端地址由前端根据当前 hostname 自动推导（localhost → `http://localhost:3000`，其他 → `window.location.origin`），一并推送给扩展

4. **不缓存用户凭证**
   - 目标网站的 Cookie 和 localStorage 仅在用户点击"采集"时提取，在内存中传递
   - 不写入 `chrome.storage` 或任何持久化存储
   - 每次采集都是即时、有意的操作

5. **CSP（Content Security Policy）兼容**
   - MV3 的 `chrome.scripting.executeScript` 在 isolated world 中注入 content-script.js，不受页面 CSP 限制
   - 极少数企业级网站可能使用 `script-src 'none'` + SRI 阻止所有脚本注入——需实测验证，但概率极低

6. **Playwright 设置 localStorage 的执行顺序**（现有逻辑，无需改动）
   - ① 创建 browserContext → ② `context.addCookies(cookies)` → ③ `page.goto(url)` 建立 origin → ④ `page.evaluate()` 注入 localStorage → ⑤ `page.reload()` 使 localStorage 在 SPA 初始化时生效 → ⑥ `waitForLoadState('networkidle')` → ⑦ `page.waitForTimeout(2000)` → ⑧ Readability 提取

7. **错误处理**
   - 扩展未配置 → popup 显示"请先在 super-admin 设置页中授权此扩展"
   - 后端不可达 → 扩展显示"无法连接到 super-admin，请检查服务是否启动"
   - 认证失败 (401) → 扩展显示"API Token 无效，请在设置中重新授权"
   - URL 提取失败 → 扩展显示"无法读取当前页面信息"
   - 采集成功 → popup 显示"已提交 (任务 #N) [查看任务]"（链接到 Web 前端 `/jobs` 页面），5 秒后自动关闭。不弹系统通知——用户可在 JobCenter 自行查看任务状态

8. **Payload 大小限制**
   - Chrome localStorage 每个 origin 上限 5MB。极端情况下可能超过 NestJS 默认 body size 限制（100KB）
   - 应对：① NestJS 配置 `app.use(json({ limit: '5mb' }))` 提升限制；② 扩展端发送前对 payload 做 hard limit（2MB），超出时截断并警告用户
   - 实际场景中 auth token 通常 < 10KB，此限制主要作为安全兜底

9. **长时间采集的 Service Worker 存活**
   - MV3 service worker 在处理中超过 5 分钟会被 Chrome 强制终止
   - 应对：扩展端 `fetch()` 设置 4 分钟 timeout；超时后显示"采集耗时较长，请在 Web 前端查看结果"
   - 实际场景中 Playwright 采集通常在 15-45 秒完成

10. **扩展代码零构建**
    - 三个纯 JS 文件 + 一个 HTML，总计约 100 行
    - 不引入 Webpack/Vite——MV3 原生 module 支持程度未知且代码量不值得
    - popup.html 内联 CSS（~50 行），不依赖任何外部 UI 库
    - **FUTURE: 扩展功能变复杂时（多 URL 批量采集、进度展示等），引入 Vite 构建并提供 TypeScript 类型检查**

11. **API Token 存储策略**
    - Token 由后端在首次启动时自动生成（`crypto.randomBytes(32).toString('hex')` → 64 字符）
    - 数据库只存 SHA-256 哈希，不存明文
    - 每次调用 `GET /api/auth/token` 生成新 raw token 并覆盖旧哈希
    - **FUTURE: 多用户时改造为列表管理（生成/命名/吊销），支持每个用户持有多个 Token**

12. **生产部署**
    - 扩展通过 `chrome://extensions/` → 开发者模式 → "加载已解压的扩展" 选择 `extension/` 目录
    - `manifest.json` 需提前配置固定 `"key"`（PEM 公钥）确保扩展 ID 稳定
    - `externally_connectable.matches` 需包含生产域名（如 `http://8.130.118.128/*`）
    - 部署文档中注明操作步骤

### 安全问题

- 扩展通过 HTTPS 与后端通信（生产环境）
- Cookie/localStorage 仅在内存中传递，不写入扩展本地存储
- 扩展不请求 `<all_urls>` 权限（取决于 Step 0 Spike 结果），仅在用户主动点击的标签页上运行
- 后端不存储原始 cookie（仅用于单次 Playwright 会话）

## What's NOT Included (MVP Scope Cut)

- ❌ Firefox/Safari 扩展
- ❌ Chrome Web Store 发布
- ❌ 扩展内直接做 Readability 提取
- ❌ 后台自动采集（无用户感知）
- ❌ 多 URL 批量采集
- ❌ 采集进度在扩展内实时展示
- ❌ sessionStorage 提取（绝大多数网站不依赖它存 auth token）
- ❌ 多用户 / 用户登录系统
- ❌ API Token 列表管理（MVP 只有一个 Token，每次覆盖）
- ❌ 系统通知（JobCenter 页面已展示完整任务状态）
